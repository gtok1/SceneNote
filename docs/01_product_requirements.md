## 1. 서비스 개요 및 핵심 가치

### 1.1 서비스 개요

SceneNote는 애니메이션, 한국 드라마, 일본 드라마, 영화를 시청하는 사용자가 자신의 감상 기록을 체계적으로 관리하고, 특정 장면에 타임라인 핀(Timeline Pin)을 남기는 모바일 앱이다.

### 1.2 핵심 가치

> **"기록이 먼저다. 검색은 도구일 뿐이다."**

SceneNote의 핵심 가치는 **사용자의 개인 감상 기록과 타임라인 핀 경험**이다. 검색 기능은 기록을 시작하기 위한 진입 도구이며, 그 자체가 목적이 아니다.

| 핵심 가치 | 설명 |
|-----------|------|
| 감상 기록 | 본 작품, 보고 싶은 작품, 진행 중인 작품을 한 곳에서 관리 |
| 타임라인 핀 | 특정 에피소드의 특정 시간대에 메모, 태그, 감정, 스포일러 여부를 기록 |
| 개인화 | 모든 데이터는 사용자 본인의 것이며, 타인과 공유되지 않음 (MVP 기준) |

### 1.3 앱 이름 및 플랫폼

- **앱 이름:** SceneNote
- **플랫폼:** iOS / Android (Expo + React Native)
- **백엔드:** Supabase (Auth + PostgreSQL + RLS + Edge Functions)
- **외부 콘텐츠 API:** TMDB, AniList, Kitsu, TVmaze

## 2. 타겟 사용자 정의

### 2.1 주요 타겟

| 구분 | 설명 |
|------|------|
| 연령대 | 18~35세 |
| 콘텐츠 소비 패턴 | 애니메이션, K-드라마, J-드라마, 영화를 주 3회 이상 시청 |
| 기록 습관 | 본 작품 목록을 메모장, 엑셀, 레터박스 등으로 관리하는 사용자 |
| 핵심 니즈 | "이 장면이 몇 화 몇 분이었지?"라는 질문을 자주 하는 사용자 |

### 2.2 사용자 페르소나

**페르소나 A — "기록형 시청자" 이수진 (27세)**
- 애니메이션과 K-드라마를 병행 시청
- 인상적인 장면마다 타임스탬프를 메모장에 따로 적어두는 습관
- "나중에 다시 보고 싶은 장면"과 "보고 싶은 작품 목록"을 따로 관리하기가 불편함

**페르소나 B — "완주형 시청자" 김태호 (31세)**
- 여러 시리즈를 동시에 시청, 어디까지 봤는지 기억하기 어려움
- 시즌/에피소드 진행률 추적이 주된 니즈
- 스포일러를 매우 싫어해서 다른 사람의 리뷰를 잘 읽지 않음

## 3. MVP 범위 정의

### 3.1 MVP 포함 기능

| 기능 | 설명 |
|------|------|
| 회원가입 / 로그인 | 이메일 + 소셜(Google, Apple) 로그인 |
| 콘텐츠 검색 | 외부 API(TMDB, AniList 등)를 통한 작품 검색 |

## 5. 콘텐츠 유형별 처리 방식

### 5.1 콘텐츠 유형 분류

| 유형 | 예시 | 특징 |
|------|------|------|
| **시리즈 (다회차)** | 애니메이션 시리즈, K-드라마, J-드라마 | 시즌 + 에피소드 구조 |
| **영화 / 단편** | 극장판, 영화, OVA 단편 | 에피소드 없음, 단일 콘텐츠 |

### 5.2 시리즈 처리

- 외부 API에서 시즌 목록과 에피소드 목록을 가져온다.
- 사용자가 라이브러리에 추가할 때 `contents`, `seasons`, `episodes` 테이블에 저장한다.
- 진행률은 `user_episode_progress` 테이블에서 에피소드 단위로 관리한다.
- 타임라인 핀은 `episode_id`를 참조하여 생성된다.
- 시즌이 없는 시리즈(예: 단일 시즌 애니)도 시즌 1로 처리한다. [확실하지 않음 — 추후 검토]

### 5.3 영화 / 단편 처리

- 에피소드 개념이 없으므로 `seasons` / `episodes` 레코드를 생성하지 않는다.
- 타임라인 핀은 콘텐츠 자체(`content_id`)를 직접 참조한다. [Tech Lead 검토 필요] 핀이 episode_id 참조를 필수로 하는 경우 영화용 가상 에피소드 레코드 생성 여부 결정 필요.
- 감상 상태는 `user_library_items`의 status 필드로만 관리한다.

### 5.4 콘텐츠 유형 판별 기준

외부 API 응답의 `media_type` 또는 `type` 필드를 기준으로 분류한다.

| API | 시리즈 식별자 | 영화 식별자 |
|-----|--------------|------------|
| TMDB | `tv` | `movie` |
| AniList | `TV`, `TV_SHORT`, `ONA`, `OVA` | `MOVIE` |
| Kitsu | `TV`, `ONA`, `OVA` | `movie` |
| TVmaze | (시리즈 전용 API) | 해당 없음 |

## 6. 외부 API 중복 결과 처리 정책

### 6.1 상황 정의

동일 작품이 TMDB와 AniList 양쪽에 등록되어 있는 경우 (예: 진격의 거인), 검색 결과에 중복으로 표시될 수 있다.

### 6.2 처리 정책

**MVP 정책: "사용자가 선택 + 출처 표시"**

1. 검색 결과는 각 API 출처별로 구분 없이 통합 표시한다.
2. 각 결과 카드에 출처 API 레이블을 표시한다 (예: "TMDB", "AniList").
3. 동일 작품이 여러 API에서 검색된 경우, 사용자가 직접 원하는 출처의 결과를 선택한다.
4. 자동 중복 제거는 MVP에서 구현하지 않는다. [Tech Lead 검토 필요] 제목 유사도 기반 중복 감지 알고리즘 구현 가능 여부 및 오탐 위험 검토.
5. 같은 작품을 두 개의 다른 API 출처로 라이브러리에 추가하는 경우, `external_id` 기반으로 중복 경고를 표시한다. [확실하지 않음 — 추후 검토]

### 6.3 canonical 레코드 정책

- MVP에서는 canonical 레코드 통합을 구현하지 않는다.
- Phase 2에서 `content_external_ids` 테이블을 활용한 cross-API 매핑을 검토한다.
- 사용자가 선택한 API 출처가 해당 콘텐츠의 primary source가 된다.

## 7. 감상 상태 전환 규칙

### 7.1 감상 상태 정의

| 상태 | 설명 | 아이콘 |
|------|------|--------|
| `wishlist` | 보고 싶음 | 북마크 |
| `watching` | 보는 중 | 재생 |
| `completed` | 완료 | 체크 |
| `dropped` | 드랍 | X |

### 7.2 상태 전환 매트릭스

| 현재 상태 | 전환 가능 상태 | 주의사항 |
|-----------|----------------|---------|
| `wishlist` | `watching`, `completed`, `dropped` | 자유 전환 |
| `watching` | `completed`, `dropped`, `wishlist` | 자유 전환 |
| `completed` | `watching` (다시 보는 중) | 진행률 초기화 여부 [확실하지 않음 — 추후 검토] |
| `dropped` | `watching`, `wishlist` | 재시작 허용 |

### 7.3 자동 상태 전환

- 시리즈의 마지막 에피소드를 시청 완료로 체크하면 `completed` 상태로 **자동 전환 여부를 사용자에게 팝업으로 확인**한다. [확실하지 않음 — 추후 검토] 자동 전환 시 UX 혼란 가능성 있음.
- 영화는 시청 완료 체크 시 `completed`로 즉시 전환한다.

### 7.4 `completed` → `watching` 전환 시 진행률 처리

- 기존 에피소드 체크 기록은 유지한다 (삭제하지 않음).
- 타임라인 핀은 전환 시 영향받지 않는다.
- 시청 회차 개념(1회차, 2회차)은 MVP에서 지원하지 않는다. Phase 2에서 검토.

## 8. MVP 출시 기준 (Definition of Done)

### 8.1 기능 완성 기준

- [ ] 사용자가 이메일 또는 소셜 계정으로 가입/로그인할 수 있다.
- [ ] 작품을 검색하여 My Library에 추가할 수 있다.
- [ ] 라이브러리에서 감상 상태를 변경할 수 있다.
- [ ] 시리즈 작품의 에피소드 단위 시청 완료를 체크할 수 있다.
- [ ] 에피소드에 타임라인 핀을 생성할 수 있다 (timestamp, 메모, 태그, 감정, 스포일러).
- [ ] 핀 목록을 timestamp_seconds 기준으로 조회할 수 있다.
- [ ] 핀을 수정하고 삭제할 수 있다.
- [ ] 태그로 핀을 필터링할 수 있다.

### 8.2 기술 완성 기준

- [ ] 모든 사용자 데이터 테이블에 RLS 정책이 적용되어 있다.
- [ ] 외부 API 키가 클라이언트에 노출되지 않는다 (Edge Functions 경유).
- [ ] 인증되지 않은 사용자는 다른 사용자의 데이터에 접근할 수 없다.
- [ ] 앱이 iOS / Android 양쪽에서 정상 빌드된다.
- [ ] 타임스탬프 초과 입력에 대한 유효성 검사가 동작한다.

### 8.3 품질 기준

- [ ] 핀 생성 플로우에서 치명적 오류(앱 크래시)가 없다.
- [ ] 검색 결과 0건 상태에서 UI가 정상 표시된다.
- [ ] 라이브러리 50개 이상 보유 시 목록이 정상 표시된다.

## 9. 개발 우선순위 정의

### P0 — 없으면 앱이 존재하지 않음

앱의 기본 골격을 이루는 기능. P0가 완성되지 않으면 다른 기능을 테스트할 수 없다.

- 인증 (가입 / 로그인)
- 콘텐츠 검색 (최소 1개 API)
- My Library에 추가 + 감상 상태 저장
- 타임라인 핀 생성 (timestamp + 메모)
- 핀 목록 조회

### P1 — MVP 핵심 가치 구현에 필수

P0 위에 올라가는 기능으로, 사용자가 SceneNote를 쓰는 이유가 되는 기능.

- 에피소드 단위 진행률 추적
- 핀 태그 / 감정 / 스포일러
- 핀 수정 / 삭제
- 태그 기반 필터링
- 감상 상태 전환
- RLS 전체 적용
- Apple / Google 소셜 로그인

### P2 — MVP 품질을 높이는 기능

일정 여유가 있을 때 포함. 없어도 앱은 동작하지만, 완성도가 올라간다.

- 검색 결과 정렬/필터
- 최근 검색어 저장
- 라이브러리 정렬 옵션
- 태그 자동완성
- 핀 생성 시 감정 추천

### P3 — 이후 Phase에서 다룰 기능

MVP 범위 외. 별도 기획 및 설계 필요.

- Highlight Reel (명장면 모음)
- 소셜 공유
- 알림
- 추천 알고리즘
- 다크 모드
- 커뮤니티 기능
